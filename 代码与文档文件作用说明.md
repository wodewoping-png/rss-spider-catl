# 代码与文档文件作用说明

更新时间：2026-05-19（UTC）

本文用于说明本仓库目录下主要代码、配置、文档和产物文件的作用。项目整体定位是：从学术期刊 RSS 自动抓取新能源、电池、碳中和、CCUS 等方向论文，补全摘要与作者信息，按周聚合，再进行中文翻译、主题分类和 Word/Excel 周报输出。

## 一、当前主生产链路

当前真正承担主要业务价值的流水线是：

```text
feeds1211.txt
  -> spider0301.py
  -> output/news_with_abstract_YYYY-MM-DD.csv
  -> weekly_aggrerate_with_abs.py
  -> output/weekly/weekly_news_with_abstract_YYYY-MM-DD.csv
  -> classify_weekly_onefile.py
  -> output/weekly/*_translated.xlsx / *_translated.docx
```

对应的 GitHub Actions 主要是：

- `.github/workflows/rss_abs.yaml`：每日抓取与每周聚合的主 workflow。
- `.github/workflows/csv_to_xlsx.yaml`：每周调用 GPT 完成分类、翻译并导出 Excel/Word。

## 二、根目录代码文件

| 文件 | 作用 | 当前定位 |
|---|---|---|
| `spider0301.py` | 当前主爬虫。读取 `feeds1211.txt`，按 UTC 抓取“昨天 + 前天”的论文条目，解析标题、链接、来源、发表日期、DOI、末位作者，并通过 RSS、网页、Crossref、Semantic Scholar、OpenAlex、PubMed 等多层策略补全摘要。对 Wiley 和 ScienceDirect 做特殊日期处理，输出 `output/news_with_abstract_YYYY-MM-DD.csv`。 | 生产脚本 |
| `weekly_aggrerate_with_abs.py` | 当前新版周聚合脚本。读取最近 7 天日报，优先读取 `news_with_abstract_*.csv`，兼容旧版 `news_*.csv`；按归一化链接或标题去重，合并更完整的标题、摘要、作者、DOI 和来源字段，输出 `output/weekly/weekly_news_with_abstract_YYYY-MM-DD.csv`。 | 生产脚本 |
| `classify_weekly_onefile.py` | 当前周报成品生成脚本。默认选择 `output/weekly/` 最新周 CSV，调用 OpenAI 翻译标题/摘要，解析 `classification.txt`，结合关键词规则、sentence-transformers embedding 和 GPT 确认进行多标签分类，输出按类别分 sheet 的 `.xlsx` 和按类别排版的 `.docx`。 | 生产脚本 |
| `classification.txt` | 周报分类体系定义。包含光伏、正极、负极、电解质、非活性材料等、其他储能器件、热能、核能、氢氨醇、产业降碳、CCUS、海底采矿、塑料回收等类别及“包括/不包括”描述。 | 生产配置 |
| `feeds1211.txt` | 当前主爬虫使用的 RSS 源列表，覆盖 Nature、ACS、Science、Wiley、Cell Press、ScienceDirect、RSC、PNAS、NSR 等学术源。 | 生产配置 |
| `requirements.txt` | Python 依赖清单，包括 feedparser、pandas、requests、playwright、transformers、torch、python-docx、openai、openpyxl、sentence-transformers 等。 | 依赖配置 |

## 三、历史、备用和测试脚本

| 文件 | 作用 | 当前定位 |
|---|---|---|
| `spider_abstract_1216.py` | `spider0301.py` 之前的含摘要爬虫版本。具备摘要补全和 Wiley 日期校验能力，但缺少当前版本中的末位作者增强等后续改动。 | 历史版本 |
| `spider_2days_1031.py` | 旧版轻量日报爬虫。读取 `feeds1011.txt`，抓取“昨天 + 前天”的 RSS 条目，主要输出基础字段，不做当前主链路的完整摘要/作者补全。 | 旧版兼容脚本 |
| `spider_yesterday_1012.py` | 更早期的单日/昨日抓取脚本，逻辑与旧版 RSS 解析、标题清洗、日期解析有关。 | 历史脚本 |
| `weekly_aggregate.py` | 旧版周聚合脚本。读取 `output/news_YYYY-MM-DD.csv`，按上一自然周聚合并按标题去重，输出 `output/weekly/news_week_YYYY-Wxx.csv`。 | 旧版兼容脚本 |
| `best_csv_to_word.py` | 备用的离线翻译 Word 导出脚本。使用 Hugging Face `Helsinki-NLP/opus-mt-en-zh` 模型把最新周 CSV 翻译为中文并导出双语 Word，也可输出 translated CSV。 | 备用导出脚本 |
| `new_tran.py` | 较早的 OpenAI 翻译与分类脚本版本。可读取周 CSV、调用 GPT 翻译和分类、输出 Excel/Word；功能已被 `classify_weekly_onefile.py` 扩展和替代。 | 历史/备用脚本 |
| `test_classify_only.py` | 只做分类、不做翻译的测试脚本。使用稳定 ID、GPT by_id 输出、缺失结果重试和 debug dump，用于验证分类逻辑和排查分类失败。 | 测试脚本 |
| `best_filter_Qiaochu_with_abstract.py` | 面向特定主题的筛选脚本。读取含摘要 CSV，先做关键词命中，再对剩余记录用 sentence-transformers 语义相似度筛选 CCUS、CO2、plastic、deep sea mining、decarbonization 等主题。 | 专项筛选脚本 |
| `best_filter_Qiaochu_with_abstract_only_word.py` | 上述主题筛选脚本的简化版本，主要基于关键词规则，不使用 embedding 语义匹配。 | 历史/简化筛选脚本 |
| `filter_Qiaochu.py` | 更早的主题筛选脚本，主要对标题进行关键词和语义匹配，输出筛选后的 CSV。 | 历史筛选脚本 |
| `merge_and_classify_filtered_news.py` | 合并 `filtered_news/` 下多份筛选结果，按日期和 sheet 规则选择数据，去重后用 sentence-transformers 将 CO2 相关文献独占分类到 CO2 电化学转化、光/光热转化、热化学转化、CO2 捕集与分离、其他。 | 专项分析脚本 |
| `reclassify_co2_news.py` | 对已有合并 CSV 做 CO2 十一类重分类，输出平铺 CSV 和按类别分 sheet 的 XLSX。默认会排除 carbonherald 链接。 | 专项分析脚本 |

## 五、GitHub Actions 工作流

| 文件 | 作用 | 备注 |
|---|---|---|
| `.github/workflows/rss_abs.yaml` | 新版主 workflow。每天 UTC 02:00 运行 `spider0301.py`，每周五 UTC 02:50 运行 `weekly_aggrerate_with_abs.py`，并把日报/周报 CSV commit 回 main。 | 当前主生产 workflow |
| `.github/workflows/csv_to_xlsx.yaml` | 每周五 UTC 03:40 运行 `classify_weekly_onefile.py -c classification.txt`，生成 `*_translated.xlsx` 和 `*_translated.docx` 并提交。 | 当前周报成品 workflow |
| `.github/workflows/rss-daily-weekly.yml` | 旧版 RSS Spider workflow。每天运行 `spider_2days_1031.py`，原本也设计为运行 `weekly_aggregate.py`。目前旧周聚合 cron 与 job `if` 条件存在不一致，周聚合基本不会实际执行。 | 旧链路，仍可能产出旧日报 |
| `.github/workflows/rss_to_word.yaml` | 手动触发的备用 Word 导出 workflow，运行 `best_csv_to_word.py` 生成离线翻译版 CSV/DOCX。 | 备用手动链路 |

## 八、数据产物目录

### `output/`

该目录保存自动抓取和周报流水线产物。

| 文件模式 | 作用 |
|---|---|
| `output/news_YYYY-MM-DD.csv` | 旧版日报产物，由 `spider_2days_1031.py` 生成，字段较少，主要用于旧链路兼容。 |
| `output/news_with_abstract_YYYY-MM-DD.csv` | 新版日报产物，由 `spider0301.py` 生成，包含标题、链接、来源、发布日期、DOI、末位作者、摘要、摘要来源等字段。 |
| `output/weekly/news_week_YYYY-Wxx.csv` | 旧版周聚合产物，由 `weekly_aggregate.py` 生成。 |
| `output/weekly/weekly_news_with_abstract_YYYY-MM-DD.csv` | 新版滚动 7 天周聚合产物，由 `weekly_aggrerate_with_abs.py` 生成。 |
| `output/weekly/*_translated.xlsx` | 分类和翻译后的 Excel 周报，通常由 `classify_weekly_onefile.py` 或旧版本脚本生成。 |
| `output/weekly/*_translated.docx` | 分类和翻译后的 Word 周报，通常由 `classify_weekly_onefile.py`、`new_tran.py` 或 `best_csv_to_word.py` 生成。 |
| `output/weekly/*_classify_only.xlsx/docx` | 仅分类测试产物，通常来自 `test_classify_only.py`。 |
| `output/empty` | 空占位文件，用于保证目录或提交结构存在。 |

### `filtered_news/`

该目录保存从日报/周报中筛选出的专题文献，以及 CO2/CCUS 方向的合并、分类、重分类和报告产物。

| 文件模式 | 作用 |
|---|---|
| `filtered_news/filtered_news_Qiaochu*.csv/xlsx` | 各批次主题筛选结果。 |
| `filtered_news/weekly_news_with_abstract_*_translated.xlsx` | 从周报成品中拷贝或筛选出的专题分析输入。 |
| `filtered_news/all_literature_merged_dedup.csv` | 多批次专题文献合并去重后的平铺 CSV，由 `merge_and_classify_filtered_news.py` 生成。 |
| `filtered_news/all_literature_classified.xlsx` | CO2 专题五类分类结果，由 `merge_and_classify_filtered_news.py` 生成。 |
| `filtered_news/all_literature_reclassified_11cats.csv/xlsx` | CO2 十一类重分类结果，由 `reclassify_co2_news.py` 生成。 |
| `filtered_news/2026_CCUS_Q1_文献分析报告.xlsx` | Q1 CCUS 文献分析 Excel 报告。 |
| `filtered_news/2026_CCUS_Q1_文献分析简报.md` | Q1 CCUS 文献分析 Markdown 简报。 |
| `filtered_news/reclassification_notes.md` | CO2/CCUS 重分类说明和备注。 |

| 文件模式 | 作用 |

| `csv/carbonherald_q1_2026_tag_counts.csv` | 标签统计数据。 |
| `csv/ccus_q1_2026_中文分析报告.xlsx` | CCUS Q1 中文分析 Excel 报告。 |

| `csv/ccus_q1_2026_*.png` | 报告使用的柱状图、饼图和月度趋势图。 |

## 十、维护时优先关注的文件

日常维护通常优先查看以下文件：

1. `spider0301.py`：日报抓取质量、摘要补全、作者补全、出版商特例。
2. `weekly_aggrerate_with_abs.py`：周聚合窗口、去重和字段合并策略。
3. `classify_weekly_onefile.py`：翻译、分类、Excel/Word 输出质量。
4. `classification.txt`：类别定义、包含/排除规则。
5. `.github/workflows/rss_abs.yaml`：主抓取和周聚合自动化。
6. `.github/workflows/csv_to_xlsx.yaml`：GPT 周报成品自动化。

历史链路和专项分析脚本仍有参考价值，但修改前应先确认它们是否还被 workflow 或人工流程实际使用。
