## 1. 项目定位

这个项目本质上是一条“**学术 RSS 自动抓取 -> 摘要补全 -> 周聚合 -> 中文翻译/分类 -> Word/Excel 输出**”的自动化流水线，面向新能源、电池、CCUS、产业降碳等技术情报跟踪。

当前仓库同时保留了：

- **当前主生产链路**：以 `spider0301.py`、`weekly_aggrerate_with_abs.py`、`classify_weekly_onefile.py` 为核心
- **旧版兼容链路**：以 `spider_2days_1031.py`、`weekly_aggregate.py` 为核心
- **专项分析链路**：以 `best_filter_Qiaochu_with_abstract.py`、`merge_and_classify_filtered_news.py`、`reclassify_co2_news.py` 为核心，面向 CCUS / CO2 主题二次分析

其中，**真正持续在 GitHub Actions 上自动运行的主链路是“新版含摘要链路”**，但旧链路仍然挂在 workflow 中，且仍在每天触发。

## 3. 当前主生产链路

### 3.1 日抓取阶段：`spider0301.py`

这是当前最重要的生产脚本。

#### 输入

- `feeds1211.txt`
- 前一日缓存文件：`output/news_with_abstract_<昨日>.csv`

#### 主要逻辑

1. 以 UTC 为准，目标时间窗是“**昨天 + 前天**”。
2. 遍历 RSS 源，抽取标题、链接、来源、发布时间、DOI。
3. 对摘要采用多级回退策略补全：
   - RSS description
   - Playwright 抓页面
   - Crossref
   - Semantic Scholar
   - OpenAlex
   - PubMed
4. 对作者做补充提取，输出 `last_author`、`last_author_source`。
5. 对特殊来源做修正：
   - **Wiley**：用 canonical 日期复核，超过最近 5 天则丢弃
   - **ScienceDirect 特定 feed**：不信任 RSS 日期，直接抓最新 N 篇，并把 `pub_date` 锚定到抓取日前一天
6. 复用昨天已有摘要的数据，减少重复请求。
7. 对前 3 天内仍无摘要的历史条目做一次重试补全。
8. 对 `must_have_abstract` 但仍没有摘要的记录做最终丢弃。

#### 输出

- `output/news_with_abstract_<UTC日期>.csv`

#### 当前字段

- `title`
- `link`
- `source`
- `published`
- `published_str`
- `pub_date`
- `doi`
- `last_author`
- `last_author_source`
- `abstract`
- `abstract_source`
- `must_have_abstract`

### 3.2 周聚合阶段：`weekly_aggrerate_with_abs.py`

这是当前新版周报汇总脚本。

#### 输入

- `output/` 下最近 7 天日报
- 优先读取 `news_with_abstract_YYYY-MM-DD.csv`
- 若不存在，兼容读取旧格式 `news_YYYY-MM-DD.csv`

#### 主要逻辑

1. 取 UTC 今天起向前滚动 7 天。
2. 读入最近 7 天日报 CSV。
3. 统一 `pub_date` 为 UTC-aware 时间。
4. 仅保留 `pub_date` 落在滚动窗口内的数据。
5. 以 `link_norm` 为主键去重；无链接时退化为标准化标题。
6. 同组内做字段补齐：
   - 标题：优先更完整、更像论文标题的版本
   - 摘要：优先更长版本
   - DOI / 来源 / 作者：非空优先
   - 发布时间：取最早值

#### 输出

- `output/weekly/weekly_news_with_abstract_<UTC日期>.csv`

### 3.3 周翻译与分类阶段：`classify_weekly_onefile.py`

这是当前自动生成 Excel/Word 周报的主脚本。

#### 输入

- 默认自动选择 `output/weekly/` 中最新的周 CSV
- 分类规则文件 `classification.txt`
- 环境变量 `OPENAI_API_KEY`

#### 分类体系

当前 `classification.txt` 定义的类别包括：

- 光伏
- 正极
- 负极
- 电解质
- 非活性材料等
- 其他储能器件
- 热能
- 核能
- 氢氨醇
- 产业降碳
- CCUS
- 海底采矿
- 塑料回收

#### 主要逻辑

1. 自动选择最新周 CSV。
2. 若未显式跳过，则调用 OpenAI：
   - 翻译英文标题到 `title_zh`
   - 翻译英文摘要到 `abstract_zh`
3. 解析 `classification.txt`，抽取每类描述、包含词、排除词。
4. 执行混合分类：
   - 规则排除
   - 规则强匹配
   - `sentence-transformers/all-MiniLM-L6-v2` embedding 候选召回
   - GPT 最终确认
5. 将结果写入：
   - 每个类别一个 sheet 的 `xlsx`
   - 分类别排版的 `docx`

#### 输出

- `output/weekly/<weekly_csv_stem>_translated.xlsx`
- `output/weekly/<weekly_csv_stem>_translated.docx`

## 4. 旧版兼容链路

### 4.1 旧日报：`spider_2days_1031.py`

这是早期简化版日报脚本。

- 数据源来自 `feeds1011.txt`
- 只抓“昨天 + 前天”
- 不做完整摘要补全

### 4.2 旧周报：`weekly_aggregate.py`

- 读取 `news_YYYY-MM-DD.csv`
- 以“上一自然周（周一到周日）”为聚合窗口
- 以标题标准化去重
- 输出 `output/weekly/news_week_<ISO周>.csv`

这套链路目前仍被旧 workflow 引用，但**不是当前质量更高的主生产链路**。

## 5. 离线 Word 导出链路

### `best_csv_to_word.py`

这个脚本不依赖 OpenAI，而是使用本地 Hugging Face 模型：

- 模型：`Helsinki-NLP/opus-mt-en-zh`
- 作用：将最新周 CSV 翻译成中英双语 CSV，并导出双语 Word

输出：

- `output/weekly/<weekly_csv_stem>_translated.csv`
- `output/weekly/<weekly_csv_stem>_translated.docx`

它和 `classify_weekly_onefile.py` 的区别是：

- `best_csv_to_word.py` 偏“离线翻译 + 顺序排版”
- `classify_weekly_onefile.py` 偏“在线 GPT 翻译 + 分类分 sheet / 分章节排版”

## 6. 专项分析链路

### 6.1 主题过滤：`best_filter_Qiaochu_with_abstract.py`

这条链路用于从日报/周报里进一步筛选特定主题文献。

逻辑分两层：

1. **关键词硬匹配**
2. **语义相似度匹配**

默认关注的主题包括：

- CCUS
- carbon capture
- carbon storage
- CO2
- plastic
- deep sea mining
- decarbonization

### 6.2 合并与独占分类：`merge_and_classify_filtered_news.py`

作用：

1. 合并 `filtered_news/` 下的 csv/xlsx
2. 对 2026-03-02 之后的数据只保留 `产业降碳` 和 `CCUS` sheet
3. 去重后将结果归入 5 类之一：
   - CO2电化学转化
   - 光或光热转化
   - 热化学转化
   - CO2捕集与分离
   - 其他

### 6.3 CO2 十一类重分类：`reclassify_co2_news.py`

这一步把 CO2 相关文献再细分为 11 类，例如：

- CO₂光转化
- CO₂热转化
- CO₂电转化
- CO₂矿化
- CO₂生物转化
- CO₂捕集与分离
- CO₂运输与封存
- 系统集成与工艺耦合
- 能源经济与LCA
- 政策与产业化
- 其他

## 7. GitHub Actions 实际运行情况

截至 **2026-05-14（UTC）**，仓库中共有 4 个 active workflow：

| Workflow | 文件 | 当前状态 | 主要用途 |
|---|---|---|---|
| `RSS Spider` | `.github/workflows/rss-daily-weekly.yml` | active，仍在每天触发 | 旧版日报 + 旧版周聚合 |
| `RSS Spider + Weekly Aggregate with Abstracts` | `.github/workflows/rss_abs.yaml` | active，当前主生产 workflow | 新版日报 + 新版周聚合 |
| `Weekly CSV to XLSX+Word (GPT)` | `.github/workflows/csv_to_xlsx.yaml` | active，周报生成主链路 | GPT 翻译 + 分类输出 |
| `Weekly CSV to Word (DOCX)` | `.github/workflows/rss_to_word.yaml` | active，但当前文件只支持手动触发 | 离线翻译 + Word 导出 |

### 7.1 当前真实在跑的主链路

从最近 workflow runs 看，当前活跃的自动链路是：

1. `RSS Spider` 每天运行一次
2. `RSS Spider + Weekly Aggregate with Abstracts` 每天运行一次
3. `RSS Spider + Weekly Aggregate with Abstracts` 在周五还有一次周聚合运行
4. `Weekly CSV to XLSX+Word (GPT)` 在周五调度运行

也就是说，**仓库当前存在两条“日报自动提交”并行运行的情况**：

- 一条旧链路产出 `news_*.csv`
- 一条新链路产出 `news_with_abstract_*.csv`

这也是 git 历史中经常出现同一天两次 `chore: daily RSS results` 提交的原因。

### 7.2 `rss_abs.yaml` 是当前主生产 workflow

它的当前配置如下：

- `0 2 * * *`：每天 UTC 02:00 跑新版日抓取
- `50 2 * * 5`：每周五 UTC 02:50 跑新版周聚合

最近实际运行记录显示：

- 2026-05-14：schedule success
- 2026-05-13：schedule success
- 2026-05-12：schedule success
- 2026-05-09：schedule success
- 2026-05-08：当天有两次成功 run

2026-05-08 的两次 run 非常关键：

- 一次是 `run-rss-spider` 成功，`run-weekly-aggregate-with-abs` skipped
- 一次是 `run-weekly-aggregate-with-abs` 成功，`run-rss-spider` skipped

这说明：

- 同一个 workflow 会被不同 cron 分别触发
- job 的 `if:` 条件能够正确区分“日报 run”和“周报 run”
- **新版周聚合链路在生产上是可用的**

### 7.3 `rss-daily-weekly.yml` 仍在运行，但周任务条件存在配置不一致

该 workflow 当前配置写的是：

- `0 1 * * *`：每天 UTC 01:00 跑旧版日抓取
- `10 1 * * 5`：每周五 UTC 01:10 跑周汇总

但是 `run-weekly-aggregate` 的 `if:` 条件写成了：

- `github.event.schedule == '10 1 * * 1'`

即：

- cron 配的是 **周五**
- 判断条件却写成 **周一**

这导致实际行为是：

- 日报 job 仍然每天正常成功
- 周五这次额外触发会发生，但对应 job 不会真正执行

最近运行证据：

- 2026-05-08 出现两次 `RSS Spider` run
- 其中一条成功执行 `run-rss-spider`
- 另一条 run 中 `run-rss-spider`、`run-weekly-aggregate` 都是 `skipped`

因此，**旧 workflow 的周聚合目前基本等于失效状态**。

### 7.4 `csv_to_xlsx.yaml` 是当前周报成品输出链路

它的 current schedule 是：

- `40 3 * * 5`：每周五 UTC 03:40

作用：

- 读取最新周 CSV
- 调用 `classify_weekly_onefile.py`
- 输出 `*_translated.xlsx` 与 `*_translated.docx`
- 自动 commit 回仓库

最近运行情况：

- 2026-04-24：schedule success
- 2026-05-01：schedule cancelled
- 2026-05-01：workflow_dispatch success
- 2026-05-08：schedule cancelled
- 2026-05-08：workflow_dispatch success

job 级状态显示：

- 2026-05-01 的 cancelled run：取消发生在 `Run classify_weekly_onefile.py` 步骤
- 2026-04-10 的 failure run：`Run classify_weekly_onefile.py` 成功，但 `Commit outputs` 步骤失败

因此可以得到两个结论：

1. **GPT 分类脚本本身多数时候是可跑通的**
2. 该 workflow 的不稳定点至少有一部分出现在“长时间运行被取消”或“产物提交回仓库”阶段

### 7.5 `rss_to_word.yaml` 当前更像备用手动链路

当前文件内容只保留了：

- `workflow_dispatch`

即现在只能手动触发。

但 GitHub 历史 runs 显示它在 2026-02-27 之前曾经有 schedule 成功记录。这说明：

- 它曾经是自动链路的一部分
- 后来被改成仅手动触发
- 目前已经不是默认周报主输出方式

## 9. 真实生产流程总结

如果只看“当前真正有业务价值、且确实在跑的主链路”，可以概括为：

### 每日

1. GitHub Actions 触发 `rss_abs.yaml`
2. `spider0301.py` 读取 `feeds1211.txt`
3. 抓取最近两天论文
4. 补摘要、补作者、处理 Wiley / ScienceDirect 特例
5. 输出 `output/news_with_abstract_<date>.csv`
6. workflow 自动 commit 回 `main`

### 每周

1. GitHub Actions 在周五触发 `rss_abs.yaml` 的周聚合 job
2. `weekly_aggrerate_with_abs.py` 合并最近 7 天日报
3. 输出 `output/weekly/weekly_news_with_abstract_<date>.csv`
4. GitHub Actions 再触发 `csv_to_xlsx.yaml`
5. `classify_weekly_onefile.py` 调用 GPT 完成翻译与分类
6. 输出 `*_translated.xlsx` 与 `*_translated.docx`
7. workflow 自动 commit 回 `main`

### 备用/历史链路

1. `rss-daily-weekly.yml` 仍每天运行旧版日报
2. `rss_to_word.yaml` 当前只支持手动触发
3. `filtered_news/` 下脚本用于专题型二次分析，不属于当前默认主流水线

## 10. 当前已知问题与维护建议

### 已知问题

1. **存在两条日报 workflow 并行运行**
   - 旧链路与新链路都在每天写仓库
   - 会造成同日两次自动提交

2. **旧 workflow 的周聚合条件写错**
   - cron 是周五
   - `if` 却判断周一
   - 导致周聚合实际不执行

3. **GPT 周报 workflow 偶发取消或提交失败**
   - 取消曾发生在分类步骤
   - 失败曾发生在 commit 步骤

4. **`rss_to_word.yaml` 与历史运行记录不完全一致**
   - 当前代码只允许手动触发
   - 历史上却曾定时执行
   - 说明这条链路的定位已经变化

### 建议的维护方向

1. 若新版链路已稳定，应考虑停用旧版 `rss-daily-weekly.yml`
2. 若保留旧 workflow，至少应修正 `run-weekly-aggregate` 的 `if` 条件
3. 对 `csv_to_xlsx.yaml` 建议重点检查：
   - 长时 GPT 调用是否容易被取消
   - 自动 commit / rebase / push 是否存在竞争
4. 建议在仓库内明确标注：
   - 哪些脚本是生产脚本
   - 哪些脚本是历史脚本
   - 哪些脚本只用于专项分析

## 11. 一句话结论

这个仓库当前的**有效主流程**已经很清晰：`spider0301.py` 负责高质量日报，`weekly_aggrerate_with_abs.py` 负责周聚合，`classify_weekly_onefile.py` 负责最终周报成品输出；但仓库里仍保留旧 workflow 和旧脚本，并且它们还在自动运行，导致当前自动化体系呈现“**主链路可用，但存在历史链路并行、配置不一致、部分周报任务偶发不稳定**”的状态。
